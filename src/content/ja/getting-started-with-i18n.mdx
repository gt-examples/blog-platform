Internationalization（略して i18n、国際化）は、アプリケーションをコードの変更なしにさまざまな言語や地域向けに適応できるよう設計するプロセスです。これは、特定のロケール向けにコンテンツを実際に調整する作業であるローカリゼーションに先立つ、基盤となるステップです。

## なぜ国際化が重要なのか

Web はグローバルです。インターネット利用者の 60%以上は、英語以外の言語での閲覧を好みます。最初から i18n を前提に構築しておけば、後から高コストなリファクタリングを避けられ、製品をはるかに大きな利用者層に届けることができます。

主なメリットは次のとおりです。

- **市場拡大** -- アプリケーションを作り直すことなく、新しい地域のユーザーにリーチできます。
- **より良いユーザー体験** -- 人は母語のコンテンツの方が、より親しみを持って深く利用してくれます。
- **SEO 上の優位性** -- ローカライズされたコンテンツは、各地域の検索エンジンでより高く評価され、検索結果でも上位に表示されやすくなります。

## 基本概念

### ロケール識別

ロケールは通常、言語コードと地域コードの組み合わせで表されます。例えば、アメリカ英語を表す `en-US` や、日本で使われる日本語を表す `ja-JP` などです。BCP 47 標準がその形式を定義しています。

### テキストの外部化

最初の具体的なステップは、コードベースからユーザー向けのすべての文字列を抽出し、外部のリソースファイルに切り出すことです。コードと翻訳リソースを分離することで、翻訳者は開発者とは独立して作業できるようになります。

```typescript
// 文字列をハードコーディングする代わりに
const greeting = "Welcome back";

// 外部化する
const greeting = t("greeting.welcome_back");
```


### ロケール対応の書式設定

数値や日付、通貨、さらには並び順までロケールごとに異なります。多くのモダンなフレームワークでは、こうした処理を自動化するための API が提供されています。

```typescript
// 各ロケールに合わせて正しくフォーマット
new Intl.NumberFormat("de-DE").format(1234.56); // "1.234,56"
new Intl.NumberFormat("en-US").format(1234.56); // "1,234.56"
```


## 実装戦略

次の手順から始めましょう。

1. コードベース全体からハードコードされた文字列とロケール依存のフォーマットを洗い出す。
2. 自分たちのスタックに適した i18n フレームワークを選定する。
3. ロケールの検出とルーティングを設定する。
4. 文字列を抽出し、翻訳ワークフローを構築する。
5. コンテンツを翻訳者に渡す前に、疑似ローカライズでテストする。

この投資はすぐに報われます。適切に国際化されたアプリケーションは、保守が容易で、複数市場へスケールさせやすくなります。
