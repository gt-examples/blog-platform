国际化（通常缩写为 i18n）是指将应用设计为在无需进行工程层面修改的情况下，即可适配不同语言和地区的过程。它是本地化之前的基础步骤，而本地化则是将内容实际针对特定语言环境进行调整的过程。

## 为什么国际化很重要

Web 是面向全球的。超过 60% 的互联网用户更喜欢使用非英语的语言浏览。通过从一开始就考虑 i18n，你可以避免后期代价高昂的重构，并让产品触达更大规模的受众。

主要收益包括：

- **拓展市场** -- 在不重建应用的前提下触达新地区的用户。
- **更佳的用户体验** -- 人们在使用母语内容时会有更高的参与度。
- **SEO 优势** -- 本地化内容在本地搜索引擎中的排名更高。

## 核心概念

### 区域设置标识

区域设置通常由语言代码和区域代码的组合来表示，例如 `en-US` 表示美式英语，`ja-JP` 表示在日本使用的日语。BCP 47 标准对其格式作了规定。

### 文本外置

第一个实际步骤，是将代码库中所有面向用户的字符串提取到外部资源文件中。通过这种分离，翻译人员就可以在不依赖开发人员的情况下独立工作。

```typescript
// 不要硬编码字符串
const greeting = "Welcome back";

// 将其外部化
const greeting = t("greeting.welcome_back");
```


### 区域设置感知的格式化

数字、日期、货币，甚至排序方式都会因区域设置而异。现代框架提供的 API 可以自动处理这些差异。

```typescript
// 为每个语言区域正确格式化
new Intl.NumberFormat("de-DE").format(1234.56); // "1.234,56"
new Intl.NumberFormat("en-US").format(1234.56); // "1,234.56"
```


## 实施策略

从以下步骤开始：

1. 审查代码库里的硬编码字符串和依赖语言环境的格式化逻辑。
2. 选择一个适合你技术栈的 i18n 框架。
3. 设置语言环境检测和路由。
4. 抽取文案并建立翻译工作流。
5. 在将内容发送给译者之前，先使用伪本地化进行测试。

这类投入会很快见效。一个国际化做得好的应用更易于维护，也更容易在不同市场上扩展。
